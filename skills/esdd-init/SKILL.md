Initialize the `feature-spec/` directory structure in the current project and auto-generate `config.yaml` with project context.

---

**Input**: No arguments required.

**Steps**

1. **Check if feature-spec already exists**

   If `feature-spec/` directory already exists:
   - If `feature-spec/config.yaml` exists, show current config and ask:
     - "Regenerate config (overwrite)" ‚Äî re-detect project context
     - "Keep existing config" ‚Äî skip
   - If `feature-spec/config.yaml` is missing, proceed to generate it
   - Ensure subdirectories exist (specs/, changes/) ‚Äî create any that are missing

   If `feature-spec/` does not exist, create the full structure.

2. **Create directory structure**

   ```bash
   mkdir -p feature-spec/specs feature-spec/changes
   ```

3. **Auto-detect project context**

   Scan the project root to detect:

   **Tech stack** ‚Äî read these files (if they exist):
   - `package.json` ‚Üí frontend framework, dependencies, node version
   - `*.csproj` / `*.sln` ‚Üí .NET version, target framework
   - `tsconfig.json` ‚Üí TypeScript config
   - `nuxt.config.ts` / `vite.config.ts` ‚Üí build tool and framework
   - `tailwind.config.*` ‚Üí CSS framework
   - `docker-compose.yml` / `Dockerfile` ‚Üí containerization
   - `go.mod` / `Cargo.toml` / `requirements.txt` ‚Üí other languages

   **Conventions** ‚Äî read these files (if they exist):
   - `CLAUDE.md` ‚Üí team standards, commit conventions, architecture patterns
   - `.eslintrc*` / `eslint.config.*` ‚Üí linting rules
   - `.prettierrc*` ‚Üí formatting rules
   - `.stylelintrc*` / `stylelint.config.*` ‚Üí style linting rules
   - `.editorconfig` ‚Üí editor settings

   **Lint commands** ‚Äî detect available lint/format tools and generate `lint_commands`:
   - Check `package.json` scripts for `lint`, `lint:fix`, `format`, `stylelint` entries
   - Check for config files: `eslint.config.*`, `.eslintrc*`, `.prettierrc*`, `.stylelintrc*`, `stylelint.config.*`
   - Check for `.csproj` / `.sln` ‚Üí `dotnet format`
   - Build the list using **fix mode** (e.g., `--fix`, `--write`) so auto-fixable issues are resolved
   - Use the project's package manager (pnpm/npm/yarn) based on lock file detection
   - Example output:
     ```yaml
     lint_commands:
       - pnpm eslint --fix .
       - pnpm prettier --write .
     ```

   **Domain** ‚Äî infer from:
   - Project name (from package.json or directory name)
   - README.md (if exists, first few paragraphs)
   - Directory structure patterns (e.g., `pages/`, `api/`, `models/`)

   **Agent roles** ‚Äî read agent definitions from `agents/` directory (if exists in the plugin installation path) to understand available roles.

4. **Generate config.yaml**

   Read the template at `skills/esdd-init/templates/config.yaml` for the reference structure and available options. Write `feature-spec/config.yaml` by filling in detected values:

   - `context:` ‚Äî populate with detected tech stack, conventions, domain
   - `lint_commands:` ‚Äî populate with detected lint/format commands (uncomment the key)
   - `commit_squash:` ‚Äî leave commented out (enabled by default; only uncomment if user wants `none`)
   - `rules:` ‚Äî populate with project-specific rules if detected (uncomment the key)
   - Add a comment: `# Auto-generated by /esdd-init on YYYY-MM-DD`

   **Guidelines for context content:**
   - Be specific: "Vue 3 + Nuxt 4 + TypeScript strict" not just "JavaScript"
   - Include architecture patterns: "Frontend: Atomic Design + Composable Pattern"
   - Include conventions: "Conventional commits", "English code comments"
   - Keep it concise ‚Äî this is read by AI every time artifacts are generated

5. **Show summary**

   ```
   ## feature-spec Initialized

   **Location:** feature-spec/
   **Config:** feature-spec/config.yaml

   ### Detected Context
   - Tech stack: [summary]
   - Conventions: [summary]
   - Domain: [summary]

   ### Directory Structure
   feature-spec/
     config.yaml          ‚Üê project context (edit to customize)
     specs/               ‚Üê accumulated main specs
     changes/             ‚Üê active changes

   ### Next Steps
   Run `/esdd-propose` to create your first change.
   ```

6. **If auto-detection is insufficient**, use **AskUserQuestion** to fill in gaps:
   - "What's the primary tech stack for this project?"
   - "Any specific architecture patterns or conventions to follow?"
   - Only ask if critical context can't be inferred

---

## Guardrails

- Never overwrite existing config without user confirmation
- Always create missing subdirectories even if feature-spec/ partially exists
- Auto-detection should be best-effort ‚Äî prefer having some context over asking too many questions
- config.yaml should be human-readable and editable
- Include the generation date as a comment in config.yaml
- If agents/ directory exists, mention available agent roles in the summary
